# Kesoku Message Data Model & Lifecycle Specification

Kesoku operates on a highly decoupled, asynchronous Pure Broker architecture. At the center of this architecture is the `Message` data model, persisted in SQLite and broadcast across active in-memory listeners via `Gateway.post()`.

To support multi-user concurrency, autonomous tool execution, reasoning transparency, and graceful interruption handling, every message in Kesoku is strictly structured around three core axes: **Role**, **Type**, and **Status**.

---

## 1. Message Attributes Matrix

### Roles (`role`)
Defines the functional entity responsible for authoring the message.
- `user`: An external human user or client interacting via a chat platform (CLI, Discord, etc.).
- `assistant`: The Kesoku autonomous AI agent (both internal reasoning thoughts and external textual responses).
- `tool`: External skill execution requests (tool calls) or the returned computation results (tool outputs).
- `system`: Internal orchestration instructions, automated followup prompts, or exception notifications.

### Types (`type`)
Categorizes the specific nature of the message content.
- `text`: Standard conversational dialogue intended for presentation or LLM ingestion.
- `thought`: Internal reasoning or chain-of-thought generated by the assistant prior to deciding on tool calls.
- `tool_call`: A requested execution of a registered skill, formatted with the target tool name and JSON arguments.
- `tool_result`: The returned execution output (or exception stack trace) resulting from an invoked skill.

### Lifecycle Statuses (`status`)
Tracks the operational state of the message within the asynchronous processing pipeline.
- `pending_agent`: Newly ingested user message queued in SQLite awaiting pickup by a session worker.
- `processing`: Actively being evaluated by an agent session worker or mid-execution within a tool call.
- `pending`: An outgoing assistant text response that has been generated but is awaiting final transmission by a chatbot adapter.
- `completed`: Successfully delivered final response on the external chat platform.
- `responded`: Successfully processed intermediate message (such as tool calls, tool outputs, thoughts, or system followups) that does not require further action.
- `interrupted`: Message processing halted or pivoted because the user submitted a newer prompt mid-turn.

---

## 2. Message Lifecycle State Machine

```
[User Inputs Message] 
       │
       ▼
(status: pending_agent) ──► [Agent SessionWorker Picks Up] 
                                   │
                                   ▼
                         (status: processing)
                                   │
           ┌───────────────────────┴───────────────────────┐
           ▼                                               ▼
[Agent Generates Thought/ToolCall]               [Agent Generates Final Text]
(status: responded)                              (status: pending)
           │                                               │
           ▼                                               ▼
[Tool Executes -> Output]                        [Chatbot Adapter Broadcasts]
(status: responded)                                        │
           │                                               ▼
           ▼                                     (status: completed)
[System Followup Prompt]
(status: responded)
           │
           └────────► (Loops back to LLM generation)
```

## 3. Complete End-to-End Execution Example

Consider a scenario where an external user asks Kesoku to perform a calculation: `"What is 25 * 40?"`

Here is the complete chronological trajectory of `Message` records created, posted, and updated during this single conversational turn:

### Step 1: User Ingestion
The user submits their query via the CLI. The CLI chatbot adapter instantiates a user message and posts it to the gateway.
```json
{
  "id": "msg_001",
  "session_id": "sess_abc",
  "sender": "User",
  "role": "user",
  "type": "text",
  "content": "What is 25 * 40?",
  "status": "pending_agent"
}
```

### Step 2: Session Worker Pickup
The Agent dispatcher detects `msg_001` (`pending_agent`) and assigns it to the `SessionWorker` for `sess_abc`. The worker updates the status in SQLite to indicate active processing:
- **Update**: `msg_001` status changes from `pending_agent` ──► `processing`.

### Step 3: Assistant Thought & Tool Call Request
The worker calls the LLM (Gemini). Gemini determines it needs to calculate the math expression. It generates an internal reasoning thought and requests the `calculator` tool. The worker posts both messages:
```json
{
  "id": "msg_002",
  "session_id": "sess_abc",
  "parent_id": "msg_001",
  "sender": "Kesoku",
  "role": "assistant",
  "type": "thought",
  "content": "I should use the calculator tool to compute 25 * 40.",
  "status": "responded"
}
```
```json
{
  "id": "msg_003",
  "session_id": "sess_abc",
  "parent_id": "msg_001",
  "sender": "Kesoku",
  "role": "tool",
  "type": "tool_call",
  "content": "Calling tool `calculator` with arguments:\n```json\n{\n  \"expression\": \"25 * 40\"\n}\n```",
  "status": "responded"
}
```

### Step 4: Tool Execution & Output
The worker executes `calculator("25 * 40")`. It captures the result (`1000.0`) and posts a tool result message:
```json
{
  "id": "msg_004",
  "session_id": "sess_abc",
  "parent_id": "msg_001",
  "sender": "calculator",
  "role": "tool",
  "type": "tool_result",
  "content": "Tool `calculator` returned:\n```\nResult of 25 * 40 = 1000.0\n```",
  "status": "responded"
}
```

### Step 5: System Followup Orchestration
To prompt the LLM to formulate the final human-friendly response, the worker synthesizes a system followup message containing the tool execution summaries and posts it:
```json
{
  "id": "msg_005",
  "session_id": "sess_abc",
  "parent_id": "msg_001",
  "sender": "System",
  "role": "system",
  "type": "text",
  "content": "User request was: What is 25 * 40?\nTool execution results:\nTool 'calculator' returned: Result of 25 * 40 = 1000.0\nPlease formulate the final response to the user based on these results.",
  "status": "responded"
}
```

### Step 6: Final Assistant Response & Completion
The worker invokes the LLM once more, providing the updated history. Gemini generates the final answer. The worker posts the final message and marks the original user message as responded:
```json
{
  "id": "msg_006",
  "session_id": "sess_abc",
  "sender": "Kesoku",
  "role": "assistant",
  "type": "text",
  "content": "25 * 40 is 1,000.",
  "status": "pending"
}
```
- **Update**: Original user message (`msg_001`) status changes from `processing` ──► `responded`.

Because `msg_006` is posted with `status = "pending"`, the registered `CLIChatbot` listener detects it, prints the beautiful rich markdown panel to the terminal, and updates the DB status upon delivery:
- **Update**: `msg_006` status changes from `pending` ──► `completed`.
